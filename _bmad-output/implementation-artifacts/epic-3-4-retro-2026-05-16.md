## Challenges

### 1. RBAC Permission Boundary Errors (Critical)

**Pattern:** Endpoints reused across stories without verifying that the target user role has the required permission.

**Evidence:**
- Epic 4 code review: 3 out of 3 HIGH findings were RBAC-related
  - Share dialog called `/api/admin/users` (requires `HR_ADMIN`) — share users only have `candidate:share`
  - CV download proxy requires `recruitment:read` — reviewers via share-token don't have this
  - No dedicated user search endpoint existed for the `candidate:share` permission level

**Root cause:** Three-layer failure:
1. **Story-level:** Permission assumptions not explicit enough in story specs
2. **Dev-level:** Endpoint reuse without permission audit against target role
3. **Review-level:** RBAC caught only in adversarial review, not earlier

**Impact:** Security boundary breaches — reviewer could receive 403 errors or, worse, access data beyond their intended scope.

### 2. XSS/Injection Recurring (Epic 3)

- JSON-LD XSS via `</script>` in vacancy data (Story 3.1)
- HTML injection in email template (Story 3.3)
- Unsafe content block casts without runtime type guards (Story 3.2)
- Occurred in 3 out of 5 core Epic 3 stories

### 3. Race Conditions (Epic 3)

- Duplicate application submissions (Story 3.3) — fixed with `$transaction` + P2002 catch
- Rapid dialog switches showing wrong candidate profile (Story 3.4) — fixed with AbortController + stale response guard

### 4. WCAG Compliance Details

- Contrast failures: `text-gray-400` at small sizes fails 4.5:1 ratio (fixed to gray-500)
- Touch targets: Tab triggers at 40px instead of 44px minimum
- GDPR footer hidden during loading/error states
- Recurring across Stories 3.1, 3.2, 3.4

## Key Insights

1. **RBAC is the #1 risk for Epic 5** — Cross-role interactions (admin configures, reviewer writes, recruiter reads) require explicit permission mapping per endpoint.

2. **Code review is non-negotiable** — Without adversarial review, the 3 HIGH RBAC bugs in Epic 4 would have shipped to production.

3. **Pattern establishment pays dividends** — Stories that follow established patterns (3.5+, 3.7–3.9) are dramatically cleaner than pattern-establishing stories (3.1, 3.3).

4. **Security concerns cluster in new interaction patterns** — XSS appeared when rendering user data in new contexts (JSON-LD, emails). RBAC appeared when crossing role boundaries. Predict where new patterns emerge and focus review there.

## Significant Discovery Alert

**RBAC permission boundary design requires attention before Epic 5.**

Epic 5 introduces the most complex cross-role interaction yet:
- **Admin** configures scorecard templates (Story 5.1)
- **Reviewer** submits evaluations via scoped view (Story 5.2) — first write operation from share-token context
- **Recruiter** views aggregated scores (Story 5.3) and compares candidates (Story 5.4)

Without the RBAC Permission Matrix and scoped-view write architecture, Epic 5 will repeat Epic 4's permission boundary mistakes at higher severity (write operations vs read-only).

**Recommendation:** Complete Action Items 1 and 2 before creating Epic 5 stories.
